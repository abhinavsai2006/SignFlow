# Requirements Document

## Introduction

This spec covers four interconnected improvements to the SignFlow e-signature platform:

1. **Premium Signature UI Redesign** — The current signature drawing canvas and post-signing certificate panel are below the quality bar of industry peers (Adobe Sign, DocuSign, Aadhaar eSign). The canvas, signing modal, and certificate display must be redesigned using the Meta design system (DESIGN-meta.md) to feel legal, trustworthy, and professional.

2. **Signature Metadata Capture** — The certificate panel shows "Unavailable" for IP Address, Location, Browser, and Device & OS. The server-side capture logic already exists in `signatureRoutes.js`, but the captured values are not being correctly propagated back to the frontend state or persisted in a way the client reads. All fields must be real-time captured and displayed.

3. **PDF Signature Embedding Bug** — When a document is finalized, the drawn or typed signature image must appear visually in the exported PDF. Currently the finalized PDF does not always contain the embedded signature. The PDF generation pipeline in `pdfService.js` must reliably embed the signature value.

4. **Storage Persistence Bug** — Documents uploaded to Cloudflare R2 become inaccessible after a user logs out and back in. The root cause is that relative R2 storage keys are stored in the database but the document list/detail endpoints do not resolve them to signed URLs before returning them to the frontend. Storage URL resolution must be fixed.

## Glossary

- **Signature_Canvas**: The HTML5 canvas element inside the signing modal where users draw their handwritten signature using the `react-signature-canvas` library.
- **Signing_Modal**: The overlay dialog that appears when a user clicks a signature placeholder field on a document, containing the canvas, type, and upload tabs.
- **Certificate_Panel**: The "Verified Electronic Signature Certificate" modal dialog that displays after a field is signed, showing signer identity, metadata, and cryptographic hash.
- **Signature_Field**: A `SignatureField` MongoDB document representing a placed placeholder that captures position, dimensions, recipient, status, signature value, and all signing metadata.
- **SignatureMetadata**: The set of fields captured at signing time: `ipAddress`, `browser`, `device`, `operatingSystem`, `location`, `isp`, `userAgent`.
- **PDF_Service**: The `backend/services/pdfService.js` module responsible for embedding signature images into PDF pages using `pdf-lib`.
- **Finalization_Pipeline**: The `POST /api/signatures/finalize` route that loads the original PDF, calls `embedSignaturesToPdf`, applies a cryptographic X.509 signature, uploads to R2, and updates the Document record.
- **R2_Storage**: Cloudflare R2 (S3-compatible) object storage where uploaded PDFs and finalized documents are stored.
- **Signed_URL**: A time-limited, pre-authenticated URL generated by the `getSignedUrl()` function in `r2Service.js` that allows the browser to directly access a private R2 object.
- **GeoIP_Service**: The external IP geolocation API (ip-api.com / ipinfo.io fallback) used server-side to resolve an IP address to a city/region/country string.
- **Meta_Design_System**: The design token system defined in `DESIGN-meta.md`, specifying colors (`#0064E0` primary, `#1C1E21` ink, etc.), typography (Optimistic VF), spacing, and component patterns.
- **Document_Editor**: The `frontend/src/components/editor/DocumentEditor.tsx` React component that hosts the PDF viewer, field placement canvas, signing modal, and certificate panel.
- **Dashboard**: The React components (`AllDocuments`, `Dashboard`, `PendingDocuments`, `CompletedDocuments`) that list documents fetched from `GET /api/docs`.

## Requirements

### Requirement 1: Premium Signature Drawing Canvas

**User Story:** As a signer, I want a smooth, professional signature drawing experience, so that my drawn signature looks high-quality and feels as trustworthy as Adobe Sign or DocuSign.

#### Acceptance Criteria

1. WHEN the Signing_Modal is opened in Draw mode, THE Signature_Canvas SHALL render with a minimum height of 200px and fill the full available modal width (minimum 400px on desktop, 100% width on mobile), providing adequate drawing space.

2. WHEN a user draws on the Signature_Canvas, THE Signature_Canvas SHALL apply velocity-smoothed ink rendering using `minWidth: 1.5` and `maxWidth: 4.0` stroke weight variation with `velocityFilterWeight: 0.6` to simulate natural pen pressure.

3. WHEN the Signing_Modal is opened, THE Signing_Modal SHALL display a baseline guide — a horizontal dashed line approximately 60% from the top of the canvas — to indicate the natural signing baseline.

4. WHEN the Signature_Canvas is empty and the user attempts to apply the signature, THE Signing_Modal SHALL prevent submission and display an inline validation message: "Please draw your signature before applying."

5. WHEN a user clears the Signature_Canvas and redraws, THE Signing_Modal SHALL preserve undo history for up to 10 strokes using the existing `drawingHistory` ref.

6. WHILE the Signing_Modal is open in Draw mode, THE Signing_Modal SHALL display three ink color selector buttons (Black `#000000`, Dark Gray `#4A4A4A`, Blue `#0064E0`) styled as 28px circular swatches using `rounded.circle` with a 2px offset ring on the active color.

7. WHEN the user selects the Type mode tab, THE Signing_Modal SHALL render a live signature preview in the selected font at a minimum 36px size against the canvas background, updating in real-time as the user types.

8. IF the user's device viewport width is less than 768px, THE Signing_Modal SHALL render the canvas at full device width with a minimum height of 160px and the mode-switcher tabs SHALL wrap to a single row.

### Requirement 2: Professional Certificate/Badge Display

**User Story:** As a document owner, I want the post-signing certificate to look as professional and trustworthy as an Aadhaar eSign certificate, so that all parties have confidence in the legal validity of the signed document.

#### Acceptance Criteria

1. WHEN a user clicks on a Signed Signature_Field, THE Certificate_Panel SHALL open and display all of the following labeled sections: Signer Name, Email Address, Verification Status, Timestamp, IP Address, Location, Browser, Device & OS, Certificate ID, Audit ID, and SHA-256 Fingerprint.

2. THE Certificate_Panel SHALL display a green verification banner at the top using `bg-emerald-50` / `border-emerald-200` containing a CheckCircle icon and the text "SignFlow Verified Signature — cryptographically verified and legally binding under ESIGN & UETA regulations."

3. WHEN all SignatureMetadata fields are populated on the Signature_Field, THE Certificate_Panel SHALL display the real captured values for IP Address, Location, Browser, and Device & OS — none of these fields SHALL display "Unavailable" for a real (non-localhost) signing event.

4. THE Certificate_Panel SHALL render the Timestamp field in the format `DD MMM YYYY • HH:MM UTC` (e.g., "07 Jun 2026 • 10:51 UTC").

5. WHEN the document has been finalized and a SHA-256 checksum is present, THE Certificate_Panel SHALL display the full hex SHA-256 fingerprint in a dark monospace code block styled with `bg-slate-900` / `text-slate-200`.

6. THE Certificate_Panel SHALL include a "Compliance Verification" section showing Trust Score, Document Integrity, Signature Integrity, Audit Trail, and Tamper Detection — all displaying "Verified" / "100%" when the field status is Signed.

7. WHERE the signing event occurred in a local development environment (IP 127.0.0.1 or ::1), THE Certificate_Panel SHALL display a yellow `AlertTriangle` banner warning the user that metadata is simulated.

### Requirement 3: Separate Size Controls for Signature and Signer Info Block

**User Story:** As a document owner, I want separate controls to independently resize the signature drawing area and the signer information block in the PDF stamp, so that I can fine-tune the layout of the embedded signature for different document types.

#### Acceptance Criteria

1. THE Document_Editor field properties panel SHALL display two independent controls for a selected Signature or Initials field: a "Signature Scale" range slider (range 25%–200%, step 25%) and a "Signer Info Scale" dropdown (Small / Medium / Large).

2. WHEN the "Signature Scale" slider is adjusted, THE Document_Editor SHALL update the `signatureScale` property of the Signature_Field and the visual preview on the canvas SHALL reflect the change immediately.

3. WHEN the "Signer Info Scale" dropdown is changed, THE Document_Editor SHALL update the `metadataScale` property of the Signature_Field, which controls the font size of the name, date, and certificate ID lines in the PDF stamp footer.

4. WHEN the PDF_Service embeds a signature with `signatureScale: 150`, THE PDF_Service SHALL render the signature drawing at 150% of its default fitted size within the top 75% of the field bounding box.

5. WHEN the PDF_Service embeds a signature with `metadataScale: 'Large'`, THE PDF_Service SHALL render the signer info text at 1.25× the base font size in the bottom 25% of the field bounding box.

### Requirement 4: Responsive Toolbar for Mobile and Desktop

**User Story:** As a user on any device, I want the document editor toolbar to adapt cleanly to my screen size, so that all essential actions are accessible without horizontal scrolling or overlap.

#### Acceptance Criteria

1. WHEN the Document_Editor is viewed on a viewport wider than 1024px (lg breakpoint), THE Document_Editor SHALL display the full desktop header with page navigation, zoom controls, Undo/Redo, Share Link, and the Finalize/Download button in a single horizontal bar.

2. WHEN the Document_Editor is viewed on a viewport narrower than 1024px, THE Document_Editor SHALL display the mobile bottom toolbar with Fields, Recipients, Zoom, Timeline, and Sign buttons, and the mobile 3-row top header.

3. WHEN the mobile bottom toolbar Sign button is tapped, THE Document_Editor SHALL trigger the same finalization flow as the desktop "Finalize & Sign" button, including the pending-fields pre-check.

4. WHEN the mobile drawer is open and the user taps outside the drawer, THE Document_Editor SHALL close the drawer without any state change to the document or signature fields.

5. IF a drag-to-pan gesture is detected on the mobile canvas (single-finger drag), THE Document_Editor SHALL scroll the PDF viewport instead of triggering field placement.

### Requirement 5: Real-Time Signature Metadata Capture

**User Story:** As a document owner, I want to see the signer's real IP address, browser, device, OS, and approximate location captured at the moment of signing, so that the audit trail is complete and legally defensible.

#### Acceptance Criteria

1. WHEN a signer submits a signature via `PUT /api/signatures/:id/sign`, THE Signature_Field SHALL be updated with `ipAddress`, `browser`, `device`, `operatingSystem`, `location`, and `isp` values captured from the HTTP request at that moment, before the response is sent.

2. WHEN the `PUT /api/signatures/:id/sign` endpoint responds, THE response body SHALL include all SignatureMetadata fields (`ipAddress`, `browser`, `device`, `operatingSystem`, `location`, `isp`, `userAgent`) alongside the updated Signature_Field data, so the frontend can update its state without an additional fetch.

3. WHEN the Document_Editor receives the sign response, THE Document_Editor SHALL update the local `signatures` state array entry for the signed field with the full metadata from the response, merging all SignatureMetadata fields into the local state.

4. IF the signer's IP address resolves to a private/loopback range (127.x.x.x, 192.168.x.x, 10.x.x.x), THEN THE GeoIP_Service SHALL return `location: 'Local Development Environment'` and `isp: 'Development Network'` without making external API calls.

5. IF the primary GeoIP_Service (ip-api.com) request fails or returns a non-success status, THEN THE GeoIP_Service SHALL fall back to the secondary service (ipinfo.io) and return the location from that response.

6. FOR ALL Signature_Field documents with `status: 'Signed'` and a non-localhost IP address, the `ipAddress`, `browser`, `device`, `operatingSystem`, and `location` fields SHALL NOT contain the string 'Unavailable'.

### Requirement 6: PDF Signature Image Embedding

**User Story:** As a document owner, I want the finalized PDF to visually contain the drawn or typed signature image in the correct position on each page, so that the signed document is visually verifiable without any additional software.

#### Acceptance Criteria

1. WHEN `generateFinalizedPdf` is called with a Document and an array of signed Signature_Fields, THE PDF_Service SHALL load the original PDF bytes from `originalFileUrl` (R2) or `originalPath` (local fallback) and embed each field's `value` onto the correct page and position.

2. WHEN a Signature_Field's `value` starts with `data:image/png`, THE PDF_Service SHALL decode the base64 data, embed the PNG using `pdfDoc.embedPng()`, and draw the image scaled to fit within the top 75% of the field's bounding box while preserving aspect ratio.

3. WHEN a Signature_Field's `value` starts with `data:image/jpeg`, THE PDF_Service SHALL embed it using `pdfDoc.embedJpg()` with the same scaling behavior.

4. WHEN a Signature_Field's `value` is a typed signature string in the format `{fontPrefix}:{text}`, THE PDF_Service SHALL render the text portion using the appropriate PDF standard font (HelveticaBoldOblique for cursive/great-vibes, TimesRomanBoldItalic for dancing-script/serif, HelveticaBoldOblique for allura/sans-serif) centered within the top 75% of the field bounding box.

5. WHEN `embedSignaturesToPdf` processes a field, THE PDF_Service SHALL skip any field where `field.value` is null, undefined, or an empty string, logging a warning but not throwing an error.

6. FOR ALL signed Signature_Fields where `value` starts with `data:image`, a round-trip test SHALL hold: `embedSignaturesToPdf` with the field SHALL produce a PDF document whose bytes, when loaded by `PDFDocument.load()`, SHALL have the same number of pages as the original PDF (no pages dropped or added by the embed step).

7. WHEN the Finalization_Pipeline saves the finalized PDF, THE Document record SHALL have `finalizedFileUrl` set to the R2 storage URL (or local path fallback) and `finalizedPath` set to the relative key, and the `status` SHALL be updated to `'Signed'`.

### Requirement 7: Storage Persistence Across Logout/Login

**User Story:** As a user, I want all my previously uploaded documents to appear in my dashboard after logging out and back in, so that my work is never lost.

#### Acceptance Criteria

1. WHEN the `GET /api/docs` endpoint returns documents to the authenticated user, THE endpoint SHALL resolve the `originalFileUrl` of each document to a valid, time-limited Signed_URL using `getSignedUrl()` before including it in the response, if the document is stored in R2_Storage.

2. WHEN the `GET /api/docs/:id` endpoint returns a single document, THE endpoint SHALL resolve both `originalFileUrl` and `finalizedFileUrl` (if present) to valid Signed_URLs before sending the response.

3. WHEN the Document_Editor calls `GET /api/docs/:id/download`, THE endpoint SHALL stream the PDF bytes by resolving the file path through `readPdfBytes()`, which SHALL call `downloadFile()` when R2_Storage is active — not rely on the raw URL being browser-accessible.

4. IF a document's `originalFileUrl` is a relative storage key (e.g., `uploads/file-12345.pdf`) without a host prefix, THE endpoint SHALL correctly extract the key and generate a Signed_URL via `getSignedUrl()` rather than returning the raw relative key.

5. FOR ALL documents owned by a user, the following round-trip property SHALL hold: uploading a document, logging out, logging back in, and calling `GET /api/docs` SHALL return that document in the list with an accessible `originalFileUrl` that resolves to the original PDF bytes.

6. WHEN a Signed_URL expires (default 3600 seconds), THE frontend SHALL re-fetch the document details to obtain a fresh Signed_URL rather than caching stale URLs across sessions.
